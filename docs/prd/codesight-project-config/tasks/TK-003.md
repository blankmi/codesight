# TK-003: Relocate jdtls data to `.codesight/lsp/` with fallback

- **Track**: Backend
- **Type**: logic
- **Modules**: `cmd/cs`, `pkg/lsp`

## Goal

Store jdtls workspace data in `<project>/.codesight/lsp/java/jdtls-data/` when `.codesight/` exists, falling back to the current `~/.codesight/jdtls-data/<hash>/` path when it does not.

- `cmd/cs/main_test.go` (add tests for new jdtls path logic)

- `pkg/lsp/lifecycle_test.go` (if lifecycle.go is changed)

## Specs

- FR-04 (LSP Data in Project Directory)
- FR-07 (Backward Compatibility)

## Data Sources & Invariants

- **Decision logic for jdtls data path**:
  1. If `<projectRoot>/.codesight/` directory exists AND is writable -> use `<projectRoot>/.codesight/lsp/java/jdtls-data/`
  2. Otherwise -> fall back to current behavior: `<stateDir>/jdtls-data/<sha256[:16]>/`
- **`.codesight/.gitignore` auto-creation**: When using the project-local path, if `.codesight/.gitignore` does not exist, create it with content `lsp/\n`. If it exists but does not contain `lsp/`, append `lsp/\n`.
- **Directory permissions**: Created directories must use `0o700`.
- **No migration**: Existing data in `~/.codesight/jdtls-data/` is not moved. Projects that run `cs init` will start using the new path; old data can be manually cleaned up.

## Implementation Notes

- Modify `jdtlsDataDir(workspaceRoot string)` to check for `.codesight/` existence first.
- Use `os.Stat()` to check `.codesight/` existence. Use a test write (create + remove temp file) or `os.MkdirAll` to verify writability.
- The `.codesight/.gitignore` logic should be a helper function reusable by TK-004's `cs init`.
- `cs clear` changes (optional LSP cleanup) are deferred ‚Äî this task only changes the data path. `cs clear` will be addressed if a follow-up epic is scoped.
