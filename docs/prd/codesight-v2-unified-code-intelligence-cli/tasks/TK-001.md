# TK-001

## Header
- Track: Extraction
- Type: logic

## Goal
Create a production-ready extraction core that can resolve one named symbol from a file or directory target with `raw` and `json` output contracts.

## Data Sources & Invariants
- Source of truth for extraction request: function input (`targetPath`, `symbol`, `format`) only.
- Source of truth for parser selection: extension/language registry in `pkg/extract/languages.go`.
- Invariants:
  - Deterministic traversal order for directory mode.
  - Supported extraction language set is exactly: `go`, `python`, `java`, `javascript`, `typescript`, `rust`, `cpp`, `xml`, `html`.
  - C (`.c`) is out-of-scope for v2 extraction requirements.
  - Directory skip list includes at least `.git`, `node_modules`, `vendor`, `__pycache__`, `.idea`, `.vscode`.
  - `raw` file mode returns symbol source only, no prefix/suffix.
  - `raw` directory mode prefixes each match with `=== file: <path> ===`.
  - `json` shape must include fields: `name`, `code`, `file_path`, `start_line`, `end_line`, `start_byte`, `end_byte`, `symbol_type`.

## Implementation Notes
- Keep extraction engine independent from Cobra and CLI concerns.
- Exact output format contracts are mandatory (including field names and directory header pattern).
- Symbol-not-found errors must include substring `symbol not found`.
- Unsupported language handling must be explicit and test-covered.
- XML/HTML support is required for parity; if additional tree-sitter grammars are needed, add them in this task.
- Backward compatibility: do not modify existing splitter behavior; extraction registry may reuse splitter language mappings but must not change search/indexing outputs.
