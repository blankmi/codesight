# TK-006: Add Java Gradle re-sync suppression policy (Phase 2)

- **Track**: Backend
- **Type**: logic
- **Modules**: `cmd/cs`, `pkg/lsp`

## Goal

Reduce Java cold-start overhead by disabling Gradle re-import on unchanged build metadata.

## Data Sources & Invariants

- **Source of truth for Java build-change detection**: workspace build files fingerprint/mtime.
- **Tracked file set (exact)**:
  - `build.gradle.kts`
  - `build.gradle`
  - `settings.gradle.kts`
  - `settings.gradle`
- **Exclusions**: Maven files (including `pom.xml`) are out of scope for this epic.
- **Initialization options invariant**:
  - Unchanged build metadata -> include `settings.java.import.gradle.enabled = false`.
  - Changed/no baseline -> do not set that override.
- **Merge invariant**: existing `settings.java.import.gradle.java.home` option remains preserved when configured.
- **Non-Java invariant**: behavior unchanged for non-Java languages.

## Implementation Notes

- Keep map-merge deterministic by key population rules; tests should decode map content, not rely on map iteration order.
- Persist the detector baseline in daemon state tied to workspace/language key.
- Edge cases to handle explicitly:
  - Missing build files.
  - Build files deleted/added between runs.

- **Coverage expectations**:

  - Changed launch: suppression flag removed.
